파라미터를 변경하기 전에 코드 레벨에서 깊이 분석해라.
**코드 안 읽고 추측하지 마라. 감으로 파라미터 건드리는 건 기도다.**

## Phase 1: 정의 찾기

1. **어디서 정의되는지.** argparse, config yaml, dataclass, 하드코딩 전부 검색.
2. **기본값은 뭔지.** 여러 곳에서 다른 기본값이면 어디가 실제로 쓰이는지 확인.
3. **타입과 범위.** int/float, 범위 제한 있는지.

```bash
# 정의 찾기
grep -rn "param_name" --include="*.py" --include="*.yaml"
```

## Phase 2: 사용 추적

**이 파라미터가 실제로 어떤 계산에 쓰이는지 전부 추적해라.**

1. 직접 사용 위치 전부 찾기 (Grep).
2. 각 위치에서 어떤 수식/로직에 들어가는지 코드 읽기.
3. 수식 안에서의 역할 파악. (예: `loss = (1-α)*L_a + α*L_b`에서 α의 역할)

## Phase 3: 영향 분석

값을 바꾸면 뭐가 달라지는지 매핑:

```markdown
| 값 | 동작 | 예상 효과 |
|----|------|-----------|
| 0 | {α=0이면 L_b가 완전히 꺼짐} | {특정 loss가 무시됨} |
| 현재값 | {현재 동작} | {현재 결과} |
| 1 | {α=1이면 L_a가 완전히 꺼짐} | {다른 loss만 남음} |
| 극단값 | {범위 밖 또는 경계} | {불안정? 발산? 무시됨?} |
```

**경계값과 불안정 구간을 특히 주의해라.** 어떤 값에서 갑자기 동작이 바뀌는지.

## Phase 4: 상호작용 분석

1. **같은 수식에 들어가는 다른 파라미터** 찾기.
2. **이 파라미터 바꾸면 다른 파라미터도 바꿔야 하는지** 확인.
3. **비율 관계** 있으면 명시. (예: lr과 batch_size는 같이 움직여야)

## Phase 5: 판단

```markdown
## 파라미터: {이름}

### 정의
- 파일: {path}:{line}
- 타입: {type}
- 기본값: {value}
- 현재값: {value}

### 코드 사용
| 위치 | 수식/로직 | 역할 |
|------|-----------|------|
| {path:line} | `loss = (1-α)*L_a + α*L_b` | blending weight |

### 값별 영향
| 값 | 동작 | 효과 |
|----|------|------|

### 상호작용
| 관련 파라미터 | 관계 |
|---------------|------|

### 판단
- 현재 값 적절한가? {근거}
- 바꾸면 뭐가 달라지나? {구체적으로}
- 권장: {바꿔라/유지해라/실험 필요} — {근거}
```

## 에스컬레이션

- 사용 경로가 너무 복잡해서 추적 불가 → 추적 가능한 범위까지만 보고, 나머지 UNVERIFIED.
- 수학적으로 불안정 구간이 있는데 현재 값이 거기 가까움 → 사용자에게 경고.
- 여러 파라미터가 얽혀있어서 단독 분석 무의미 → 관련 파라미터 전부 묶어서 보고.

## 규칙

- **코드 안 읽고 추측하지 마라.** 반드시 실제 사용 위치를 확인.
- **모든 주장에 file:line 인용.**
- **"보통 이 값을 쓴다" 금지.** 이 프로젝트 코드에서 어떻게 쓰이는지만.
- **확인 안 된 건 [UNVERIFIED].**
