프로젝트 파일 구조를 점검해라.
**Iron Law: 자동 삭제 절대 금지. 모든 정리는 사용자 승인 후에만.**

## Phase 1: 구조 파악

1. **`project-structure.md` 읽기** (cleaning 스킬 폴더에 있으면). 정상 구조 기준.
2. **실제 파일 구조 스캔.** 전체 디렉토리 트리 파악.
3. **차이 식별.** 기준과 실제의 차이.

## Phase 2: 문제 탐색

| 항목 | 찾는 것 | 찾는 법 |
|------|---------|---------|
| 쓰레기 | `__pycache__/`, `.pyc`, `.DS_Store`, `*.bak`, `*.tmp` | `Glob("**/__pycache__")`, `Glob("**/*.pyc")` |
| 빈 디렉토리 | 파일 없는 폴더 | `find . -type d -empty` |
| 중복 | 같은 내용 다른 경로 | 파일명 + 크기 비교, 필요시 내용 비교 |
| 잘못된 위치 | 구조 기준에 안 맞는 파일 | scripts/ 밖의 .py, configs/ 밖의 .yaml |
| 고아 | 아무 데서도 import/참조 안 됨 | `Grep("import {module}")` 결과 0 |
| 크기 이상 | 비정상적으로 큰 파일 | `find . -size +10M` |
| 오래된 파일 | 오랫동안 수정 안 된 활성 영역 파일 | `find . -mtime +90` (scripts/, libs/) |

## Phase 3: 분류 + 보고

발견한 문제를 세 카테고리로 분류:

```markdown
## 점검 결과

### 삭제 권장
| 경로 | 이유 | 크기 |
|------|------|------|
| `__pycache__/` | 빌드 캐시 | 2.3MB |

### 이동 권장
| 현재 | 이동할 곳 | 이유 |
|------|-----------|------|
| `scripts/old_test.py` | `legacy/scripts/` | 미사용 |

### 아카이브 권장
| 경로 | 이동할 곳 | 이유 |
|------|-----------|------|
| `logs/old_exp/` | `logs/archive/` | 오래된 실험 |

### 문제 없음
{깨끗한 영역}
```

## 보호 구역 — 절대 건드리지 마라

| 경로 | 이유 |
|------|------|
| `.git/` | git 내부 |
| `context/archive/` | 영구 보존 |
| `logs/` | 삭제 금지. archive 이동만. |
| `papers/` | 논문 원본 보존 |
| `legacy/` | 참조용 보존 |

## 에스컬레이션

- 판단 애매한 파일 → "문제 없음"으로 분류. 잘못 지우는 것보다 안 지우는 게 낫다.
- 보호 구역 안에서 문제 발견 → 보고만. 직접 건드리지 마라.
- 프로젝트 구조가 `project-structure.md`와 많이 다름 → 구조 자체 업데이트 제안.

## 규칙

- **자동 삭제 절대 금지.** 보고만 하고 사용자 승인 기다려라.
- **`logs/`는 삭제 금지.** `logs/archive/`로 이동만.
- **확신 없으면 "문제 없음".** 의심스러우면 남겨라.
- **결과를 테이블로 정리.** 사용자가 한눈에 보고 판단할 수 있게.
