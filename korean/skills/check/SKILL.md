# /check — 코드 전체 점검

코드를 잘게 쪼개서 에이전트 여러 개한테 병렬로 보내라. 혼자 눈으로 훑지 마라.
핵심은 부분부분 정확하게 점검하는 거다. 추측으로 "괜찮아 보인다" 금지. 근거 있는 검증만.

## 사용법

```
/check <파일경로>
```

---

## Phase 1: 코드 읽기

점검 전에 대상 코드를 먼저 읽고 이해해라. 안 읽고 에이전트한테 던지지 마라.

1. **대상 파일 읽기.** 전체 구조, 핵심 로직 파악.
2. **관련 파일 확인.** import하는 모듈, 호출하는 함수, 참조하는 config.
3. **논문 참조 확인.** `papers/` 에 관련 논문 summary가 있으면 읽어서 수식 대조 준비.

---

## Phase 2: 분할 + 병렬 점검

코드를 기능 단위로 쪼갠다. 함수별, 모듈별, 로직 블록별로. 각 조각마다 전담 에이전트를 배정해서 동시에 보내라. 한 에이전트가 전체를 보는 게 아니라, 여러 에이전트가 각자 맡은 부분을 정밀하게 본다. 개수 아끼지 마라 — 10개 보내도 된다.

### 수학 검증

math-verifier 에이전트 + Codex twin 병렬 실행.

### 점검 항목

| 우선순위 | 관점 | 항목 | 찾는 것 |
|----------|------|------|---------|
| CRITICAL | 수학 | 부호 오류 | gradient ascent vs descent, loss 부호 반전 |
| CRITICAL | 수학 | 빠진 항 | 논문 수식에 있는데 코드에 없는 term |
| CRITICAL | 수학 | detach 누락 | target 계산에서 gradient가 새는 곳 |
| CRITICAL | 연구 | 코드 ≠ 주장 | 논문/문서에 쓴 것과 코드가 실제로 다른 곳 |
| CRITICAL | 연구 | 데이터 누수 | test→train 혼입, 미래 정보가 현재 step에 |
| HIGH | 수학 | mean vs sum | batch size에 따라 결과 달라지는 곳 |
| HIGH | 수학 | broadcasting | 차원이 자동 확장되면서 의도와 다르게 계산 |
| HIGH | 수학 | discount/GAE | γ, λ 적용 순서, 방향 |
| HIGH | 연구 | 비교 공정성 | 베이스라인과 조건이 동일한가. 우리만 유리한 세팅 아닌가 |
| HIGH | 연구 | 단일 시드 의존 | 하나의 시드 결과를 일반화하고 있지 않은가. 최소 3 seeds, 이상적으로 5+ |
| HIGH | 연구 | 베이스라인 부재 | 클래식 방법, 최근 SOTA, 단순하지만 강한 baseline (linear probe 등) 중 빠진 게 없는가 |
| HIGH | 연구 | ablation 설계 | 핵심 컴포넌트를 하나씩 빼는 ablation이 있는가. 각 ablation이 의미 있게 다른가 |
| MEDIUM | 수학 | 수치 불안정 | `log(0)`, `exp(큰수)`, `0으로 나누기` |
| MEDIUM | 수학 | dtype | float32/64 혼용, 정밀도 손실 |
| MEDIUM | 연구 | 재현 가능성 | 시드, 설정, 환경이 전부 기록/고정돼 있는가 |
| MEDIUM | 연구 | 리소스 보고 | wall-clock time, 메모리 사용량이 정확도와 함께 보고되는가 |
| MEDIUM | 연구 | 변수 통제 | 비교할 때 딱 하나만 바꿨는가. 여러 개 동시에 바꾸면 뭐 때문인지 모른다 |

### 검증 방법

- **논문 수식과 코드 한 줄씩 대조.** 변수명 매핑 테이블 만들어라.
- **작은 입력으로 손계산 vs 코드 출력 비교.** 숫자가 맞는지 직접 확인.
- **gradient 흐름 추적.** `.detach()`, `with torch.no_grad()` 위치가 맞는지.

---

### 성능 체크

perf-optimizer 에이전트 + Codex twin 병렬 실행.

### 점검 항목

| 우선순위 | 항목 | 찾는 것 |
|----------|------|---------|
| HIGH | 불필요한 복사 | `.clone()`, `.cpu()↔.cuda()` 루프 안에서 반복 |
| HIGH | 루프 안 텐서 생성 | 매 step 새로 만드는 텐서 (밖으로 빼야) |
| HIGH | CPU-GPU 핑퐁 | 데이터가 device 사이를 왔다 갔다 |
| MEDIUM | 벡터화 가능 | for 루프를 텐서 연산으로 바꿀 수 있는 곳 |
| MEDIUM | 메모리 누수 | 리스트에 텐서 계속 쌓기, history 안 비우기 |
| LOW | 미사용 계산 | 계산하지만 결과를 안 쓰는 코드 |

## Phase 3: 안전성 체크

별도 에이전트 없이 직접 점검.

1. **NaN/Inf 체크.** 중요 텐서에 shape, NaN, Inf assert가 있는지.
2. **시드 고정.** `torch`, `numpy`, `random`, CUDA 시드가 전부 고정되어 있는지.
3. **에러 처리.** 파일 I/O, 네트워크, GPU 메모리 — 터졌을 때 graceful하게 처리하는지.
4. **하드코딩.** GPU index, 서버 경로, 매직 넘버 하드코딩 여부.

---

## Phase 4: 교차 검증 + 보고

1. **에이전트 결과 비교.** math-verifier와 Codex가 같은 걸 찾았는지, 다른 걸 찾았는지.
   - **불일치 = 가장 중요한 신호.** 무시하지 말고 파고들어라.
2. **보고 양식은 `report-template.md` 참고.**

## 에스컬레이션

- 논문 수식이 모호해서 검증 불가 → 사용자에게 보고. 추측으로 넘어가지 마라.
- 에이전트 3번 다른 답 → 멈추고 사용자한테 올려라.
- 성능 이슈가 구조적 (파이프라인 전체 리팩토링 필요) → 보고만. 임의로 리팩토링 시작하지 마라.

## 상태 보고

- **PASS** — 수학/성능/안전 전부 문제 없음.
- **WARN** — minor 이슈 있음. 목록 명시.
- **FAIL** — critical 이슈 발견. 즉시 수정 필요. 위치와 수정안 명시.
- **BLOCKED** — 검증 불가 (논문 없음, 코드 이해 불가 등). 사유 명시.
