# /fresh — 세션 시작

`/clear` 후 컨텍스트를 로드해서 이전 세션을 이어간다.
이 스킬이 실패하면 세션 전체가 맥락 없이 시작된다. 절대 대충 하지 마라.

---

## Phase 0: GPT Pro 응답 확인 (가장 먼저)

`context/gpt-pro/` 없으면 건너뛴다.

1. `context/gpt-pro/inbox/*.md` glob
2. **응답 있으면:**
   - 매칭되는 `outbox/` 요청도 읽기
   - 요청 내용과 대조해서 핵심 인사이트 2-3줄 추출
   - 우리 상황에 적용 가능한 부분과 아닌 부분을 구분해서 정리
   - 둘 다 `context/gpt-pro/archive/`로 이동
   - 인사이트를 메모리에 보관 — Phase 1 컨텍스트 로드 후 통합
3. **응답 없으면:** 건너뛴다. outbox는 보지 마라.

**실패 조건:** inbox 파일이 깨져있거나 읽을 수 없으면 → 사용자에게 보고하고 다음으로 넘어간다. archive 이동 실패 시 → 사용자에게 경고.

---

## Phase 1: 컨텍스트 로드

**반드시 이 순서대로.** 파일마다 실제로 열어서 읽어라. 존재 확인만 하고 넘어가지 마라.

1. **`CLAUDE.md`** — 프로젝트가 뭔지, 지금 어디까지 왔는지, 기술 스택, 제약사항.
   - 이게 없으면 프로젝트를 이해할 수 없다. 없으면 사용자에게 보고.

2. **`context/short_term.md`** — 지난 세션에서 뭘 했고, 다음에 뭘 해야 하는지.
   - 이걸 읽고 바로 이어서 작업할 수 있어야 한다. 모호하면 사용자에게 확인.
   - **`context/archive/short_term_*.md` 중 가장 최근 1개도 읽어라.** 이전 세션 맥락까지 파악하면 연속성이 더 좋다. 없으면 건너뛴다.

3. **`context/long_term.md`** — 확정된 결론, 방향, 왜 이 방향인지.
   - 현재 방향의 근거를 이해해야 한다. 이해 안 되면 질문해라.

4. **`context/exp_log.md`** (있으면) — 최근 실험 결과 확인.
   - 마지막 실험이 뭐였는지, 결론이 뭐였는지 파악.

**검증:** 로드한 내용끼리 충돌이 없는지 확인. CLAUDE.md에 "Phase A 진행 중"인데 short_term에 "Phase B 시작"이면 → 사용자에게 확인.

## Phase 3: 보고

사용자에게 현재 상태를 **구체적으로** 보고한다. "컨텍스트 로드 완료" 같은 빈 말 금지.

양식은 `report-template.md` 참고.

**핵심:** 이 보고를 읽은 사용자가 아무것도 추가로 읽지 않아도 바로 "그래 이거 하자"라고 말할 수 있어야 한다.

---

## 상태 보고

- **DONE** — 모든 컨텍스트 로드 완료, 이전 세션과 연속성 확보됨.
- **DONE_WITH_CONCERNS** — 로드했지만 충돌/모호한 부분 있음. 구체적으로 명시.
- **BLOCKED** — 핵심 파일(CLAUDE.md 또는 short_term.md) 없음. 작업 시작 불가.

---

## 주의

- `/clear` 먼저 하고 `/fresh` 해라. 안 그러면 이전 대화 잔재가 섞인다.
- short_term.md가 3일 이상 안 갱신됐으면 → stale 경고 출력.
- long_term.md가 2주 이상 안 갱신됐으면 → stale 경고 출력.
