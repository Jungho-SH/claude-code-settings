# /team — Agent Teams 편성

Agent Teams를 사용해서 독립 세션 팀을 편성한다.
**subagent(`Agent` 도구)와 다르다.** Teams는 각 팀원이 독립 세션으로 살아있고, 서로 메시지를 주고받고, 토론하면서 방향을 좁힐 수 있다.

- **subagent**: 시키면 결과 가져오고 끝. 일방통행.
- **Agent Teams**: 독립 세션이 유지되고, 팀원끼리 소통하고, 공유 태스크로 조율한다.

## 사용법

```
/team "왜 학습 후반에 더 넘어지는지 분석"
/team "reward 설계 방향 토론"
/team "FiLM vs cerebellar 비교 분석"
```

---

## Phase 0: 작업 분석 + 팀 구성 결정

1. **작업을 분석해라.** 이 작업에 몇 명이 필요한지, 어떤 관점이 필요한지 파악.
2. **팀원 역할 설계.** 각 팀원에게 명확한 역할과 관점을 부여. 같은 관점을 가진 팀원은 의미 없다.
3. **토론이 필요한지 판단.** 정답이 있는 작업이면 subagent로 충분하다. 방향 선택, 설계 결정, 트레이드오프 분석처럼 토론이 필요한 작업에 Teams를 써라.

### 팀 구성 예시

| 작업 | 팀원 구성 | 왜 이렇게 |
|------|-----------|-----------|
| 방향 선택 | 찬성파 + 반대파 + 중립 분석가 | 한쪽으로 쏠리지 않게 |
| 디버깅 | 수학 전문가 + 성능 전문가 + 코드 분석가 | 다른 관점에서 원인 탐색 |
| 실험 설계 | 설계자 + 비평가 + 선행연구 조사자 | 빈틈 없는 설계 |
| 논문 분석 | 방법론 분석가 + 우리 프로젝트 비교 전문가 | 위협 정확히 파악 |

---

## Phase 1: 팀 생성

```
TeamCreate(
  team_name: "{주제}-team",
  description: "{작업 설명}"
)
```

---

## Phase 2: 태스크 생성 + 팀원 투입

### 태스크 먼저 만들어라

각 팀원이 할 일을 `TaskCreate`로 명확하게 정의. 팀원이 자기 태스크를 claim해서 작업한다.

### 팀원 프롬프트 — 이게 제일 중요하다

각 팀원한테 보내는 프롬프트에 반드시 포함:

```markdown
# 팀: {팀 이름}
# 네 역할: {구체적 역할과 관점}
# 너의 태스크: {뭘 해야 하는지}

## 맥락
{프로젝트가 뭔지 2-3줄}
{지금 뭘 하고 있고 왜 이 팀을 띄웠는지}
{다른 팀원들은 어떤 역할인지}

## 네가 해야 할 것
{구체적 결과물}

## 규칙
- 실제 코드/데이터를 읽고 근거를 대라. 추측 금지.
- 다른 팀원과 의견이 다르면 근거를 들어서 반박해라. 그냥 동의하지 마라.
- 확신 없으면 확신 없다고 명시.
```

**"대충 알겠지" 하고 던지지 마라.** 팀원은 우리 대화를 모른다.

### 투입

```
Agent(
  team_name: "{팀 이름}",
  name: "{역할명}",
  prompt: "{위 프롬프트}",
  run_in_background: true
)
```

독립적인 팀원은 전부 동시에 보내라.

## Phase 3: 조율 + 토론

Teams의 핵심. subagent와의 결정적 차이.

1. **팀원 결과를 공유해라.** `SendMessage`로 팀원 A의 발견을 팀원 B한테 전달.
2. **의견 충돌을 유도해라.** 팀원들이 전부 동의하면 오히려 의심해라. 일부러 반론을 요청해라.
3. **토론을 통해 수렴해라.** 양쪽 근거를 비교하고, 어느 쪽이 더 강한지 팀원들이 서로 판단하게 해라.
4. **리더로서 방향을 잡아라.** 토론이 수렴 안 되면 네가 판단해라. 또는 사용자에게 올려라.

## Phase 4: 종합 보고

```markdown
## 팀 결과: {작업 설명}

### 팀 구성
| 팀원 | 역할 | 관점 |
|------|------|------|
| {이름} | {역할} | {어떤 관점에서 봤는지} |

### 핵심 발견
1. {가장 중요한 발견}
2. {두 번째}

### 토론 요약 (있으면)
- {쟁점}: {팀원 A}는 {주장} (근거: ...), {팀원 B}는 {주장} (근거: ...)
- **결론:** {어떻게 수렴했는지, 또는 미결인지}

### 권장 다음 행동
1. {구체적 행동}

### 신뢰도: HIGH / MEDIUM / LOW
```

## Phase 5: 정리

```
TeamDelete()
```

**팀 끝나면 반드시 정리.** 고아 팀 남기지 마라.

## 언제 Teams vs Subagent

| 상황 | 뭘 쓰나 |
|------|---------|
| 정답이 있는 작업 (검색, 검증, 분석) | Subagent (`Agent` 도구) |
| 방향 선택, 설계 결정, 트레이드오프 | **Teams** (`/team`) |
| 토론/반론이 필요한 작업 | **Teams** |
| 빠르게 결과만 필요 | Subagent |
| 여러 관점에서 깊이 파고들어야 | **Teams** |

---

## 상태 보고

- **DONE** — 팀 완료, 종합 보고 작성, 팀 정리됨.
- **DONE_WITH_CONCERNS** — 완료했지만 미결 쟁점 있음. 명시.
- **BLOCKED** — 팀원 다수 실패. 사유 명시.
- **NEEDS_CONTEXT** — 정보 부족. 필요한 것 명시.
