# /cleaning — 프로젝트 파일 정리

프로젝트 파일 구조를 `project-structure.md` 기준으로 점검하고 정리한다.
**절대로 자동 삭제하지 마라.** 모든 정리는 사용자 승인 후에만.

---

## Phase 1: 현재 상태 파악

1. `project-structure.md`를 읽어서 정상 구조를 파악한다.
2. 실제 파일 구조를 스캔한다 (`find`, `ls -R` 등).
3. 정상 구조와 비교해서 차이를 식별한다.

---

## Phase 2: 문제 탐색

cleaner 에이전트를 백그라운드로 실행. 아래 항목을 전부 점검:

| 항목 | 찾는 것 | 예시 |
|------|---------|------|
| 쓰레기 | `__pycache__/`, `.pyc`, `.DS_Store`, `*.bak`, `*.tmp` | 빌드/캐시 잔해 |
| 빈 디렉토리 | 파일 없는 폴더 | 삭제된 모듈의 빈 껍데기 |
| 중복 | 같은 내용의 파일이 다른 경로에 | 복사 붙여넣기 잔재 |
| 잘못된 위치 | `project-structure.md` 기준에 안 맞는 파일 | scripts/ 밖의 .py 파일 |
| 고아 파일 | 아무 데서도 import/참조 안 되는 모듈 | 옛날 쓰다 버린 유틸리티 |
| 크기 이상 | 비정상적으로 큰 파일 | 실수로 커밋된 데이터/모델 |

---

## Phase 3: 보고

결과를 테이블로 정리해서 사용자에게 보여준다.

```
## 정리 보고

### 삭제 권장
| 경로 | 이유 | 크기 |
|------|------|------|
| __pycache__/ | 빌드 캐시 | 2.3MB |

### 이동 권장
| 현재 경로 | 이동 경로 | 이유 |
|-----------|-----------|------|
| scripts/old_test.py | legacy/ | 미사용 |

### 아카이브 권장 (logs/ 포함)
| 경로 | 이동 경로 | 이유 |
|------|-----------|------|
| logs/old_exp/ | logs/archive/old_exp/ | 오래된 실험 |

### 문제 없음
{깨끗한 영역 명시}
```

---

## Phase 4: 실행

1. **사용자한테 "진행할까?" 물어봐라.** 승인 없이 절대 실행하지 마라.
2. 승인받은 항목만 실행.
3. 실행 후 결과 보고 — 뭘 했는지, 뭘 안 했는지.

---

## 절대 규칙

- **`logs/`는 절대 삭제 금지.** `logs/archive/`로 이동만. (`11-logs.md` 참고)
- **`context/`는 함부로 건드리지 마라.** archive 이동만.
- **`.git/`은 절대 건드리지 마라.**
- **확신 없으면 "문제 없음"으로 보고해라.** 잘못 지우는 것보다 안 지우는 게 낫다.

---

## 상태 보고

- **DONE** — 점검 완료, 정리 실행됨.
- **DONE_WITH_CONCERNS** — 정리했지만 판단 애매한 파일 있음. 명시.
- **BLOCKED** — 파일 접근 불가 등. 사유 명시.
