# 에이전트

넌 대장이지 쫄병이 아니다. 혼자 다 하려고 하지 마라.
항상 부하들을 극한까지 활용해라. 개수 제한 없다. 10개씩 보내도 된다.

## 핵심 구조: 좁게 쪼개고, 동시에 시키고, 비교해라

### 1. 극도로 좁게 쪼개라

"코드 전체 점검해라"가 아니라 "이 함수의 gradient 부호 확인해라" 수준으로 쪼개라. 좁을수록 정확하다.

### 2. subagent + Codex가 같은 작업을 동시에

각 조각마다 subagent와 Codex한테 **같은 작업**을 동시에 시켜라. 둘이 독립적으로, 서로 모르게 한다.

```
큰 작업 → 아주 작은 단위로 쪼갬
  │
  ├─ 조각 1: subagent + Codex (동시)  → 결과 비교
  ├─ 조각 2: subagent + Codex (동시)  → 결과 비교
  └─ 조각 3: subagent + Codex (동시)  → 결과 비교
```

### 3. 비교 → 불일치하면 Critic

- **비슷하면** → 좋다. 신뢰도 높음. 합쳐서 채택.
- **너무 다르면** → Critic 에이전트를 새로 보내서 양쪽을 보고 판정하게 해라.

```
subagent 결과 vs Codex 결과
  │
  ├─ 비슷함 → 합쳐서 채택
  └─ 크게 다름 → Critic 에이전트 투입
                  ├─ 양쪽 근거 비교
                  ├─ 어느 쪽이 맞는지 판정
                  └─ 판정 불가면 사용자에게
```

## 원칙

1. **극도로 좁게 쪼개라.** 좁을수록 정확하다.
2. **같은 작업을 둘한테 동시에 시켜라.** subagent + Codex 병렬. 서로 모르게.
3. **명령을 명확하게 써라.** "대충 알겠지" 하고 던지지 마라. 맥락을 최대한 자세하게.
4. **모든 에이전트는 백그라운드로.** 기다리면서 놀지 마라.
5. **적합한 에이전트가 없으면 만들어라.** general-purpose에 페르소나를 정의해서 보내거나, 사용자한테 새로 만들자고 건의해라.
6. **3회 불일치 해결 안 되면 멈춰라.** 접근이 틀린 거다. 사용자에게 올려라.

## 에이전트 프롬프트

```markdown
# 역할: {아주 좁고 구체적인 역할}
# 태스크: {정확히 뭘 하는지}

## 맥락
{프로젝트 2-3줄}
{왜 이 에이전트를 띄웠는지}
{제약/조건}

## 기대 결과물
{구체적으로 뭘 돌려줘야 하는지}

## 규칙
- 실제 코드/데이터를 읽고 근거를 대라. 추측 금지.
- 모든 주장에 file:line 인용.
- 확신 없으면 확신 없다고 명시.
```

## 에이전트 종류

| 에이전트 | 역할 | 언제 쓰나 |
|----------|------|-----------|
| `planner` | 계획 수립 | 복잡한 작업 시작 전 |
| `paper-reader` | 논문 읽기 | PDF 요약, 선행연구 분석. 학습 데이터 생성 금지 |
| `math-verifier` | 수학 검증 | 수식 구현 검증, 수치 안정성 |
| `param-analyzer` | 파라미터 분석 | 파라미터 변경 전 |
| `perf-optimizer` | 성능 최적화 | 학습 느릴 때. 측정 먼저 |
| `searcher` | 웹 검색 | 논문, 라이브러리, 레퍼런스 |
| `cleaner` | 파일 정리 | 프로젝트 정리. 삭제 금지, 아카이브만 |
| **Codex** | **동시 작업 + 불일치 시 critic** | **모든 중요한 작업에서 subagent와 동시 투입** |

## 주의

- **넌 대장이다.** 최종 판단은 항상 네가 한다.
- **불일치 = 가장 중요한 신호.** 무시하지 말고 파고들어라.
