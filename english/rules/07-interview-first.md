# Interview First

Don't start until you understand what you're starting.
The cheapest way to undo a wrong assumption is to find it before you act on it.

Most failed work fails at the input, not the execution.
Vague start → wrong direction → wasted hours/days → start over.
Five minutes of interview saves the whole loop.

## Trigger

Run `/interview` before:
- New experiment (any seed/sweep/run)
- Non-trivial code change
- Theory/proof work
- External communication (collaborator, reviewer, advisor)
- Adopting a new concept/library/tool
- Multi-step plan ("what should we do next")

## Skip For

- Trivial fixes (typo, rename, obvious one-liner)
- Continuing already-interviewed work in the same session
- Explicit override ("just do it"). Then run a 30-second version: goal sentence + outcome scenarios + load-bearing assumption.

## Checklist Before Acting

1. **Goal in one sentence.** If you can't articulate it, you don't have it yet. Push for clarity.
2. **All outcomes informative.** Success means X, failure means Y, null means Z. If "failure = no info", you're praying, not testing.
3. **Load-bearing assumption identified.** Which assumption, if wrong, kills the work? Can it be tested first, separately?
4. **Decision tree visible.** Not just step 1 — the whole arc. "If X then Y, if A then B."
5. **Unfamiliar parts understood.** A polite "okay" without engagement is not understanding. Probe with: "How would you apply this to our X?"

## Six Interview Disciplines

1. **One question at a time.** Multiple at once = shallow batched answers.
2. **End with a question.** Until closure is reached, never trail off.
3. **Recommend an answer with each question.** Don't ask cold.
4. **Read code/data/docs first.** If it's findable, find it before asking.
5. **Sharpen fuzzy terms immediately.** Pin down "works", "good", "stable" to numbers.
6. **Stress-test with concrete scenarios.** Abstractions hide bugs.

## When in Doubt

Run it. Five minutes spent here is cheaper than the wrong-path work it prevents.
